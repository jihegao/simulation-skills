# Simulation Skills Version 0.1 Provider 契约

状态：Workstream 0 provider 与联合契约已正式收口

正式 pin：Git tag `contracts-v0.1.0`；联合 bundle digest
`sha256:6160830cf1f8dfd1699fd97c66ca437de41beaf3795e53b9d09d2d3fe1b00fb6`

## 所有权

本仓权威发布以下闭合 Draft 2020-12 契约：

- `simulation.model_spec@0.1.0`：薄 Modeling IR envelope、边界、时间语义、带单位参数与指标、方法 payload ref；
- `simulation.result_set@0.1.0`：标准化指标、单位、精确 Model/Experiment/Simulation Run ref、诊断和 `draft_unreviewed` claim；
- `simulation.domain_pack@0.1.0`：pack version、Scenario template、method contract、默认 Experiment template、指标和 adapter requirement；
- `simulation.des_method_payload@0.1.0`：Version 0.1 队列/资源/维修所需的窄 DES payload；
- `simulation.abm_method_payload@0.1.0`：只覆盖交通路网跨域 fixture 所需的窄 ABM payload，不是通用 ABM ontology。

Workbench 继续拥有 Project、Scenario、Experiment、Workflow/Simulation Run、Verification、
Finding、Review Record、Artifact 和 Reproducibility Manifest。provider 不发布这些对象的
schema，也不创建其权威 revision 或 digest。

## Digest 和 revision

- JSON object ref 的 `digest` 始终是完整对象的 RFC 8785 canonical JSON SHA-256；对象内部不保存自指 digest。
- Domain Pack descriptor 的 `manifest_digest` 同样是完整 Domain Pack 对象的 RFC 8785 digest。
- export `documents[].digest` 是对应文件原始 bytes 的 SHA-256。
- export `provider.source_tree_digest` 是排序后完整 `documents` 数组的 RFC 8785 digest，不包含 export manifest 自身或 Git commit。
- `revision` 从 1 开始；revision 1 必须使用 `parent_revision: null`，后续 revision 必须直接指向上一 revision。
- 未知必需 extension、未知 major、错误 revision、错误或 stale digest 均 fail closed。

## Export

固定 export root：

```text
simulation_skills_contracts/conformance/v0_1/
  export-manifest.json
  adapter-manifest.json
  schemas/
  fixtures/valid/
  fixtures/invalid/
  cases/
    warehouse-queueing/case.json
    equipment-maintenance/case.json
    traffic-intersection/case.json
```

`export-manifest.json` 对每份 document 记录 owner、contract、schema version、kind、byte
digest、schema ref、validity、case ID、validation layer 和 expected failure code。构建脚本
重复运行必须产生完全一致的 bytes。

`joint-bundle.lock.json` 固定双仓 Workstream 0 正式契约；它与 Workbench 正式收口报告中的
lock 字节一致。Git tag 固定仓库提交，bundle digest 固定跨仓机器契约，两者必须同时保留。
该 pin 不授权真实 runtime，也不改变 fake adapter 的 claim 边界。

每个 case 是 `simulation.conformance_case@0.1.0` provider fragment：包含 Domain Pack、
Model Spec、对应 method payload、无 digest 的 Workbench logical required connection 和
非持久化 `expected_result_template`。它不伪造 Project、Scenario、Experiment 或 Simulation
Run ref。联合 builder 绑定 Workbench 权威对象后，才能物化 schema-valid Result Set 并计算
完整对象 digest。独立 `fixtures/valid/result-set.json` 只验证 Result Set schema shape，其中
外部 ref 是语法示例，不是联合图真源。

## Fake adapter

安装后入口为：

```text
simulation-skills-fake-adapter
```

它支持 Workbench `simulation.adapter_protocol@0.1` 的 `describe` 和 `execute`。`execute`
只接受 `equipment-maintenance` contract fixture，要求固定 seed，并要求 Workbench 在
`input_snapshot.simulation_run_ref` 中提供权威 pending Simulation Run ref。adapter 原样绑定
该 ref，产出完整 `simulation.result_set` JSON Artifact、full-object JCS ref、Artifact byte
digest 和 execution manifest。它不 import Workbench、不依赖相邻源码树路径、不写产品状态，
也不产生 `behavior_checked`。

Warehouse queueing 和 traffic intersection 是 schema/graph 组合 fixture，不表示 fake adapter
已执行或验证这两个场景。交通 case 使用 `agent_based_network` 与窄 ABM payload，防止跨域
检查退化为第三个 DES fixture。

## 真实 warehouse queue adapter

Workstream 1 的首个真实 runtime 以独立安装入口提供：

```text
simulation-skills-warehouse-des-adapter
```

其 logical adapter ID 为 `simulation-skills.simpy.warehouse-des`，adapter version 为
`0.1.0`。它只声明 `warehouse-queueing` Domain Pack、`discrete_event` 方法族和单次
`experiment.run`，不实现 sweep、comparison、Finding 或 export。

真实 manifest 位于
`simulation_skills_contracts/adapters/warehouse_des/adapter-manifest.json`，作为 wheel package
data 独立安装。它不进入 `simulation_skills_contracts/conformance/v0_1/export-manifest.json`，
因此不会改变 `contracts-v0.1.0` tag 所固定的 fake manifest、联合 bundle 或正式 digest。

`input_snapshot` 使用闭合、自包含 shape，包含：

- `seed` 和精确 `simulation_run_ref`；
- 与顶层 ref 绑定的 `model_spec_digest`、`experiment_digest`，以及精确
  `method_payload_digest`；
- Controller 提供的稳定 `result_created_at`；
- `hour` 时间单位和正数 horizon；
- exponential arrival 的正数 mean interarrival；
- 正整数 resource capacity；
- 与该 resource 绑定的 triangular service minimum、mode 和 maximum；
- 精确的 `completed_jobs`、`average_wait` requested metrics。

任何未知字段、错误 digest、非法单位、不支持的 distribution、错误 resource 绑定、非法
三角分布参数或 claim 晋升都会在写 response/Artifact 前失败。adapter 不读取 Workbench
storage；Workbench 必须先从已验证对象图生成 snapshot，再通过 subprocess 协议传入。

实际模型使用 SimPy 资源队列，并从一个显式 seed 确定性派生 arrival 与 service 两个独立
随机流。horizon 为 exclusive；`completed_jobs` 只计算 horizon 前完成的 job，
`average_wait` 计算 horizon 前已开始服务的 job。相同完整 request 在新的 invocation root
中产生 byte-identical Result Set 和相同 RFC 8785 ref；response timing 记录真实执行时间，
不属于 byte-identical 结果承诺。

结果始终为 `draft_unreviewed`，并明确声明：单次执行不证明真实仓库有效性，且未执行
校准、敏感性分析、comparison 或因果检验。当前本地调用继续沿用 Workbench 接受的
`macos_26_5_1_local_rc` 边界；这不改变 `production_sandbox_proven=false`。

## 真实 Equipment Maintenance adapter

Equipment Maintenance Domain Pack 的真实 provider runtime 以独立安装入口提供：

```text
simulation-skills-equipment-maintenance-adapter
```

其 logical adapter ID 为 `simulation-skills.simpy.equipment-maintenance`，adapter version
为 `0.1.0`。它只声明 `equipment-maintenance` Domain Pack、`discrete_event` 方法族和
`experiment.run`；manifest 位于
`simulation_skills_contracts/adapters/equipment_maintenance/adapter-manifest.json`，不进入
冻结的 Workstream 0 export。

闭合 `input_snapshot` 精确绑定 seed、Simulation Run、Model Spec、method payload、
Experiment digest、Domain Pack 的 `pack_id` / `pack_version` / installed manifest digest，
并携带资产数、正数 horizon、exponential MTBF、正整数维修资源容量、triangular repair
参数和固定的 `availability` / `unplanned_downtime` 指标请求。provider 不读取 Workbench
或 Domain Pack storage；未知字段、stale 顶层 digest、不支持的 pack/version、错误随机
分布、非法参数或 claim 晋升均在写可信输出前失败。

真实 SimPy 模型为多资产故障—FIFO 排队—维修—恢复循环。显式 seed 为每项资产派生独立
failure 与 repair 随机流；horizon 为 exclusive。`unplanned_downtime` 是所有资产在 horizon
内的停机小时总和，`availability` 为 `1 - downtime / (asset_count * horizon)`。相同完整
request 的 Result Set bytes 与 RFC 8785 ref 可确定性重放；response timing 不属于重放承诺。

Result Set optional extension 原样回显三字段 Domain Pack binding；闭合 execution manifest
仍只承载协议与 runtime 证据，不增加 Workbench-owned binding 字段。结果始终为
`draft_unreviewed`，诊断明确标注 `behavior_validation=not_evaluated` 和
`domain_certification=not_evaluated`。运行成功只证明这份声明模型已执行，不证明行为有效、
领域认证、校准、comparison、策略建议或因果结论。

## 验证

```bash
python3 scripts/build_contract_export.py
python3 -m unittest tests.test_v0_1_provider_contracts -v
python3 -m unittest tests.test_v0_1_fake_adapter -v
python3 -m unittest tests.test_v0_1_warehouse_des_adapter -v
python3 -m unittest tests.test_v0_1_equipment_maintenance_adapter -v
python3 -m unittest tests.test_v0_1_clean_install -v
python3 -m unittest discover -s tests -v
```

clean-install 测试从 wheel 安装到仓库外临时 venv，清除 `PYTHONPATH`，验证 package data、
console entrypoint、`describe` 和 `execute`。Workbench consumer 已用相同 export digest 组装
联合 bundle，并在 `macos_26_5_1_local_rc` 范围完成隔离验收；该证据使用已弃用的
`sandbox-exec` 且明确 `production_sandbox_proven=false`，不能替代生产 sandbox 决策。
该残余风险不阻塞本地 Workstream 1 开发，但继续阻塞任何生产或线上 adapter 执行声明。

- 通用 maintenance ontology、参数优化或策略推荐；
- Equipment Maintenance sweep、comparison、Finding、review 或 export；
- 通用 simulation ontology；
- 自动生成可信 Verification、Finding 或 Reproducibility Manifest；
- 根据进程退出码或 Artifact 存在自动晋升 claim。
